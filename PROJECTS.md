# 프로젝트 정리 초안

이 문서는 김보민의 진행 프로젝트를 **Data-Centric Robotics Engineer** 포트폴리오 관점에서 정리하기 위한 초안이다.

아래 내용은 대화에서 확인된 사실을 바탕으로 한 1차 구조화이며, 프로젝트별 세부 내용은 김보민에게 확인한 뒤 확정한다. 특히 어려웠던 점, 해결 과정, 강조하고 싶은 로직, 실패 사례, 직접 기여한 부분은 임의로 작성하지 않는다.

각 프로젝트는 다음 기준으로 정리한다.

- 문제: 어떤 현장 문제 또는 학습 과제를 다루었는가
- 데이터: 어떤 입력, 센서, 이미지, 로그, 학습 데이터를 사용했는가
- 실험: 어떤 기술적 시도와 구현을 했는가
- 결과: 무엇을 확인했고 어떤 역량을 증명했는가
- 증거: 코드, 사진, 영상, 데이터셋, 실험 로그 등 연결 가능한 자료

## 1. Arduino 뽀모도로 타이머

### 분류
- 핵심 축: Robotics AI
- 세부 역량: 임베디드 기초, 센서/액추에이터 제어, 사용자 인터페이스 구현
- 포트폴리오 역할: 하드웨어와 소프트웨어를 연결하는 기초 구현 경험

### 문제
학습이나 작업 시간을 일정한 주기로 관리할 수 있는 물리적 타이머를 Arduino 기반으로 구현한다.

이 프로젝트는 단순 앱이 아니라 버튼, 디스플레이, 부저 등 하드웨어 입력과 출력을 직접 제어하면서 소프트웨어 로직이 실제 장치 동작으로 이어지는 과정을 경험하는 데 의미가 있다.

### 데이터
- 사용자 입력: 버튼 또는 스위치 입력
- 출력 상태: 타이머 진행 상태, 알림 상태, 표시 장치 상태
- 시간 데이터: 작업 시간, 휴식 시간, 반복 주기

### 실험
- Arduino 코드로 뽀모도로 시간 흐름을 구현
- 하드웨어 입력을 받아 타이머 시작, 정지, 초기화 동작 구성
- 디스플레이나 LED, 부저 등 출력 장치를 통해 현재 상태를 사용자에게 전달

### 결과
- 마이크로컨트롤러에서 시간 기반 로직을 구현하는 방법을 학습
- 하드웨어 입력과 소프트웨어 상태 관리가 연결되는 구조를 경험
- 사용자에게 동작 상태를 명확하게 전달하는 임베디드 UI의 중요성을 이해

### 증거
- Arduino 코드
- 회로 또는 제작 사진
- 작동 사진/영상
- 부품 목록과 회로 구성 메모

### 웹 포트폴리오 요약 문장
Arduino 기반 뽀모도로 타이머를 제작하며 버튼 입력, 시간 제어 로직, 출력 장치 제어를 연결해 하드웨어와 소프트웨어가 함께 동작하는 구조를 구현했다.

## 2. Arduino 라인트레이서

### 분류
- 핵심 축: Robotics AI
- 세부 역량: 센서 기반 제어, 모터 제어, 로봇 주행 기초
- 포트폴리오 역할: 센서 입력을 실제 움직임으로 변환하는 로봇 제어 입문 프로젝트

### 문제
라인트레이서 로봇이 바닥의 선을 인식하고, 센서 값에 따라 주행 방향을 조정하도록 구현한다.

이 프로젝트는 로봇이 환경을 센서로 읽고, 그 입력을 기반으로 제어 명령을 만들어 실제 움직임으로 연결하는 기본 구조를 이해하는 데 의미가 있다.

### 데이터
- 라인 센서 입력값
- 모터 출력 상태
- 주행 중 성공/실패 상황
- 라인 이탈, 곡선 구간, 속도 변화에 따른 반응

### 실험
- 라인 센서를 통해 바닥 선 감지
- 센서 값에 따라 좌우 모터 속도 또는 방향 제어
- 직선, 곡선, 이탈 상황에서 제어 로직 조정
- 실제 주행 사진 또는 영상으로 동작 확인

### 결과
- 센서 입력을 제어 로직으로 변환하는 기본 구조를 학습
- 실제 하드웨어에서는 센서 노이즈, 바닥 상태, 조명, 속도에 따라 동작이 달라진다는 점을 경험
- 로봇이 현장에서 실패하는 원인이 코드뿐 아니라 센서, 환경, 제어 파라미터에도 있다는 관점을 얻음

### 증거
- Arduino 코드
- 로봇 제작 사진
- 주행 테스트 사진/영상
- 센서 배치와 제어 로직 메모

### 웹 포트폴리오 요약 문장
Arduino 라인트레이서를 구현하며 센서 입력을 모터 제어로 연결했고, 실제 환경에서 로봇 주행이 센서 상태와 제어 파라미터에 크게 영향을 받는다는 점을 학습했다.

## 3. Jetson AGX Orin 환경설정 및 YOLOv8 종이컵 탐지

### 분류
- 핵심 축: Field Data, Robotics AI
- 세부 역량: 데이터 수집, 라벨링, 데이터 증강, 객체 탐지 모델 학습, 엣지 AI 환경 이해
- 포트폴리오 역할: 직접 데이터를 만들고 학습시켜 탐지 결과까지 확인한 데이터 중심 AI 프로젝트

### 문제
AI 모델이 특정 물체를 안정적으로 탐지하려면 목적에 맞는 데이터를 직접 수집하고 라벨링해야 한다.

이 프로젝트에서는 종이컵 이미지를 촬영하고 라벨링한 뒤 YOLOv8 모델을 학습시켜, 테스트 이미지에서 종이컵을 탐지할 수 있는지 확인했다.

### 데이터
- 직접 촬영한 종이컵 이미지
- 객체 탐지를 위한 라벨링 데이터
- 데이터 증강을 통해 확장한 학습 데이터
- 테스트 이미지

### 실험
- Jetson AGX Orin 환경설정
- Ubuntu 환경에서 Python 기반 AI 개발 환경 구성
- 종이컵 이미지 촬영 및 라벨링
- 데이터 증강 적용
- Google Colab에서 YOLOv8 학습
- 테스트 이미지에서 종이컵 탐지 성능 확인

### 결과
- 직접 구축한 데이터셋으로 YOLOv8 객체 탐지 모델을 학습
- 테스트 이미지에서 종이컵 탐지 성공
- 데이터 수집, 라벨링, 증강, 학습, 테스트까지 객체 탐지 파이프라인을 경험
- 모델 성능은 코드보다 데이터 품질과 수집 조건에 크게 영향을 받는다는 점을 학습

### 증거
- 종이컵 원본 이미지
- 라벨링 데이터
- 데이터 증강 결과
- Google Colab 학습 노트북
- 학습 로그와 결과 이미지
- 테스트 이미지 탐지 결과

### 웹 포트폴리오 요약 문장
직접 촬영한 종이컵 데이터를 라벨링하고 증강한 뒤 YOLOv8을 학습시켜 테스트 이미지에서 탐지를 성공시켰으며, 데이터 수집부터 모델 검증까지 객체 탐지 파이프라인을 경험했다.

## 4. Jetson AGX Orin YOLOv8 TensorRT 변환 및 Depth Camera 실시간 탐지

### 분류
- 핵심 축: Robotics AI, Field Data, Sim2Real
- 세부 역량: 엣지 AI 추론 최적화, TensorRT, CUDA, Depth Camera, 실시간 객체 탐지
- 포트폴리오 역할: AI 모델 출력을 실제 로봇/자율주행 시스템의 센서 입력과 연결한 대표 프로젝트

### 문제
로봇이나 자율주행 시스템에서 객체 탐지 모델은 학습 정확도만으로 충분하지 않다.

실제 센서 입력을 받아 실시간으로 탐지하려면 모델을 하드웨어 환경에 맞게 최적화하고, 카메라 입력과 추론 코드를 안정적으로 연결해야 한다.

이 프로젝트에서는 Seeed Wiki사의 캐리어 보드를 사용하는 Jetson AGX Orin 환경에서 JetPack, Python, pip, numpy, CUDA, TensorRT, Ultralytics 버전이 서로 맞아야 실제 학습과 추론이 가능하다는 점이 가장 큰 문제였다.

### 데이터
- 자율주행 자동차에 연결된 Depth Camera 입력
- 실시간 영상 프레임
- YOLOv8 기존 모델의 탐지 결과
- 프레임 처리 속도 정보

### 실험
- Ultralytics에서 YOLOv8 기존 모델 import
- YOLOv8 모델을 TensorRT 기반 `.engine` 파일로 변환
- Depth Camera를 Python 코드로 연결
- 노트북 상에서 실시간 탐지 결과 확인
- CUDA를 사용해 30 FPS 이상 프레임 처리 확인

### 어려웠던 점
- 가장 많이 부딪힌 영역은 코드 로직 자체보다 **버전 호환성**이었다.
- Seeed Wiki사의 캐리어 보드를 사용하는 환경이라 JetPack을 별도로 커맨드 창에서 설치하고 설정해야 했다.
- JetPack과 연동되는 Python, numpy, pip, CUDA, TensorRT, Ultralytics 버전을 맞추기 위해 여러 번 설치, 삭제, 재설정을 반복했다.
- 처음에는 가상환경 없이 `sudo` 명령어로 시스템 패키지를 수정하다가 환경이 꼬였고, 결국 OS를 초기화한 경험이 있었다.
- AI의 도움으로 코드를 작성하고 결과를 확인하는 것은 가능했지만, 단순히 복사해서 실행하는 것이 아니라 용어와 구조를 이해하는 과정이 어려웠다.

### 해결 과정
- OS 초기화 경험 이후 가상환경과 백업의 중요성을 체감했다.
- 이후에는 프로젝트 환경을 최대한 분리하고, 위험한 시스템 변경을 줄이는 방향으로 작업했다.
- 에러가 발생했을 때는 코드만 보지 않고 버전, 파일 경로, 실행 환경, 패키지 의존성을 함께 확인했다.
- AI의 도움을 받더라도 프로젝트 상황을 정확히 설명해야 결과가 좋아진다는 점을 느꼈고, 이를 위해 내가 먼저 구조를 이해하려고 했다.
- 코드를 복사해서 실행하는 데서 끝내지 않고 한 줄씩 따라 작성하며 파일 경로와 실행 흐름을 이해하려고 했다.

### 학습한 개념
- 딥러닝 모델 학습이 단순 실행이 아니라 데이터, 학습 설정, 평가 지표가 함께 맞물리는 과정임을 이해했다.
- `epochs`, `mAP50`, `cls_loss`, `box_loss` 같은 학습 지표와 손실 개념을 접했다.
- `batch`, `optimizer`, `imgsz` 같은 설정값이 학습 결과와 속도에 영향을 준다는 점을 알게 되었다.
- 학습은 정답 버튼을 누르는 과정이 아니라 여러 요소의 조화를 찾아가는 과정이라는 점을 배웠다.

### 강조하고 싶은 판단
- Jetson AGX Orin은 CPU와 GPU 메모리 구조가 통합된 엣지 AI 보드이기 때문에, 일반 PC처럼 모델을 실행하는 것보다 하드웨어에 맞는 최적화가 중요하다고 판단했다.
- YOLOv8 모델을 그대로 사용하는 데서 멈추지 않고 TensorRT를 통해 `.engine` 형식으로 변환해 Jetson 환경에 맞춘 추론 최적화를 시도했다.
- Depth Camera를 자율주행차의 실제 센서 입력으로 연결하고, Python3 코드로 노트북에서 실시간 탐지 결과를 확인했다.
- 사소해 보이는 최적화 하나가 하드웨어와 소프트웨어 연결에서는 큰 차이를 만들 수 있고, 실제 환경에서는 그 차이가 더 크게 드러난다는 점을 체감했다.

### 결과
- YOLOv8 모델을 TensorRT `.engine` 형식으로 변환해 추론 최적화 흐름을 경험
- Depth Camera의 실제 영상 입력을 Python 코드와 연결해 탐지 동작을 확인
- CUDA 기반 환경에서 30 FPS 이상 실시간 탐지가 가능함을 확인
- AI 모델이 실제 로봇 시스템에 들어가기 위해서는 모델 변환, 하드웨어 가속, 센서 연결, 프레임 처리 속도가 함께 고려되어야 함을 학습
- 버전 호환성과 실행 환경 관리가 엣지 AI 프로젝트의 핵심 리스크라는 점을 경험
- 가상환경, 백업, 의존성 관리의 중요성을 실제 실패와 복구 과정을 통해 학습

### 증거
- Python 탐지 코드
- TensorRT `.engine` 변환 기록
- Depth Camera 연결 사진/영상
- 실시간 탐지 화면 캡처
- FPS 측정 결과
- Jetson AGX Orin 환경설정 기록

### 웹 포트폴리오 요약 문장
Seeed 캐리어 보드 기반 Jetson AGX Orin 환경에서 버전 호환성 문제를 해결하며 YOLOv8 모델을 TensorRT `.engine`으로 변환했고, Depth Camera 입력과 연결해 CUDA 기반 30 FPS 이상 실시간 탐지를 확인했다. 이 과정을 통해 엣지 AI에서는 모델 성능뿐 아니라 환경설정, 의존성 관리, 하드웨어 최적화가 실제 동작을 좌우한다는 점을 배웠다.
